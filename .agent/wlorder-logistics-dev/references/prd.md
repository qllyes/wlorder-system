# 物流发货管理系统 — 产品需求规格 (PRD)

## 1. 项目背景

3人物流小团队，需一套轻量级、无服务器依赖的 Python 物流管理系统。核心目标：打通 **客户下单 → 调度派车/外包 → 现场装货确认 → 财务核算** 的完整 O2O 闭环。通过内网穿透 + 移动端扫码，让外部司机通过手机与内部系统交互。

## 2. 用户角色

| 角色 | 终端 | 职责 |
|---|---|---|
| 调度员/文员 | PC | 录入订单、派发发货单（整车/零单）、补录第三方物流、登记费用、查看看板、手工变更整车状态为"已订车" |
| 车队司机 | Mobile | 扫码填报身份与车辆资质、确认装货发车 |
| 管理层 | PC | 查看利润核算报表与发货汇总（可由调度员兼任） |

## 3. 核心业务流转

```
客户订单 → 调度员录入【原始订单】
                ↓
         调度分拣（人工判断）
           ↙          ↘
    分支A: 整车         分支B: 零单
    (公司物流)          (外包第三方)
        ↓                  ↓
  生成发货单            生成发货单
  状态=未订车           状态=待填写
        ↓                  ↓
  调度员手工改为        文员补录物流信息
  状态=已订车           (公司名+运单号)
  (联系好司机后)        ※ 多个零单可合单
        ↓                  ↓
  生成加密二维码        状态→已发货
  司机扫码填报
  确认发车
        ↓
  状态→已发货
        ↘              ↙
       费用核算（物流费 - 实际成本 = 利润）
              ↓
         看板统计汇总
```

### 3.1 整车分支状态流（更新）

```
未订车 ──(调度员手工操作)──→ 已订车 ──(司机扫码确认)──→ 已发货
```

- **未订车**：发货单刚创建，尚未联系到司机
- **已订车**：调度员已联系好司机，手工在系统中将状态改为"已订车"。此时生成加密二维码供司机扫码
- **已发货**：司机现场扫码完成填报后自动变更

### 3.2 零单合单发货

多个零单可能由同一个司机/物流车辆承运，需支持**合单发货**：
- 调度员可将多个零单组合关联到同一批次（`batch_id`）
- 合单后的零单在列表中有明显的分组标识，方便识别哪些零单属于同一趟运输
- 合单操作不影响每个零单的独立费用核算

## 4. 功能模块

### 4.1 订单管理
- 字段：订单编号(自动)、客户名称、货物名称/品类、数量(件)、收货地址、创建时间、状态
- 交互：单行快捷录入；列表一键转入调度分发

### 4.2 发货调度（核心枢纽）
- **整车派发**：
  - 状态色标区分生命周期（红=未订车 / 橙=已订车 / 绿=已发货）
  - 调度员可手工将"未订车"变更为"已订车"（按钮操作）
  - 仅"已订车"状态才展示二维码入口（防止未确认司机即扫码）
- **零单补录**：弹窗表单，必填第三方物流公司 + 运单号
- **零单合单**：支持勾选多个零单进行合单，生成批次号

### 4.3 司机扫码确认端 (Mobile H5)
- 免登录、即扫即用
- 大字体、醒目按钮，去除 PC 端复杂布局
- Token 鉴权防伪 + 防重复提交（已发货 → 展示成功页）
- 表单：姓名、身份证、手机号、车牌号、车型(下拉选择)

### 4.4 发货单打印

- 支持按模板打印发货单（整车/零单通用）
- **模板内容**：发货单号、订单号、客户名称、货物名称、数量、收货地址、发货类型、创建时间
- **整车额外**：司机姓名、车牌号、车型
- **零单额外**：第三方物流公司、运单号
- **安全限制**：打印模板中**禁止包含**"实际成本"和"利润"等内部保密字段
- **交互**：在发货单列表的操作列提供"打印"按钮，点击后弹出打印预览窗口

### 4.5 费用与利润核算
- 对外字段：客户物流费
- 对内字段：实际成本价（保密，不对外展示/打印）
- 自动计算：单票利润 = 对外物流费 - 对内成本

### 4.6 数据看板
- 当日新增订单量
- 积压未发车数（整车"未订车"+"已订车" + 零单"待填写"）
- 当日已发货数量趋势
- 列表明细 + 分类统计卡片

## 5. 状态机

### Orders 表状态
- `待发货` → 初始化
- `已派发` → 已分配为发货单

### Shipments 表状态
- `未订车` → 整车专属，刚创建发货单
- `已订车` → 整车专属，调度员手工确认已联系好司机（**新增**）
- `待填写` → 零单专属，文员未补录
- `已发货` → 终态（整车=司机扫码确认 / 零单=文员填单）

## 6. UI 设计要求

### 6.1 整体设计哲学

- **现代化权威感**：采用专业级 UI 设计，参考企业级 SaaS 产品风格（如 Notion、Linear、Vercel Dashboard）
- **视觉层次清晰**：通过卡片阴影、色彩对比、字号层级建立信息优先级
- **交互反馈即时**：所有操作提供视觉反馈（Loading、Toast、状态变色过渡动画）
- **响应式适配**：PC 端宽屏充分利用空间，Mobile 端大按钮、大字体、触控友好

### 6.2 配色与排版

- **主色调**：专业蓝（#1677FF 或类似），传递可信赖感
- **辅助色**：状态色系（成功绿、警告橙、危险红、信息灰）
- **背景**：浅灰底（#F5F5F5）+ 白色卡片，避免纯白刺眼
- **字体**：系统原生字体栈，标题加粗突出，正文 14-16px 舒适阅读

### 6.3 组件规范

- **表格**：斑马纹、悬停高亮、固定表头、操作列吸顶
- **按钮**：主操作用实心填充按钮，次要操作用描边/文字按钮，禁用态灰化
- **状态标签**：使用 Chip/Badge 色标，颜色与状态语义强绑定
- **弹窗**：居中模态，背景蒙版，表单项间距适中，必填项星号标识
- **空状态**：无数据时显示插图 + 引导文案，而非空白

## 7. 技术选型

| 项目 | 选型 | 理由 |
|---|---|---|
| 语言 | Python 3.10+ | 团队现有经验 |
| 前后端 | NiceGUI | Vue3 渲染，PC/Mobile 同构，数据轮询刷新灵活 |
| 数据库 | SQLite + aiosqlite | 无服务器轻量，满足 10 万量级单机业务 |
| 二维码 | qrcode (python-qrcode) | 纯 Python 生成 |
| 网络 | cpolar / ngrok | 内网穿透，零运维成本 |
