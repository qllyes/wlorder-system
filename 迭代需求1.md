- # 订单管理系统 UI/UX 与交互重构开发需求文档 (DRD)
  
  **文档版本**: V1.0  
  
  **相关规范**: 遵循仓库中 `.agent/ux-interaction-designer/SKILL.md` 与 `.agent/wlorder-logistics-dev/SKILL.md`  
  
  **核心技术栈**: Python, NiceGUI, Quasar (Material Design), Tailwind CSS, SQLite  
- ## 1. 背景与目标 (Background & Objectives)
  
  **当前痛点**：  
- **视觉层级混乱**：订单明细页头部元素堆砌（返回按钮与明细信息同处一行），缺乏重点，不符合视觉层级规范。
- **操作路径冗余**：订单明细修改采用“上方展示 + 下方表单”的分离模式，违背了 UX 设计中的“就近原则 (Proximity)”，增加了用户的认知和操作负担。
- **缺乏SaaS质感**：新增订单页表单扁平、冗长，缺乏空间留白与模块化设计，交互体验停留在传统后台管理系统水平，未能达到现代顶级 SaaS 系统的标准。
  
  **重构目标**：  
  
  利用 NiceGUI 与底层 Quasar 框架的特性，对“订单明细页”和“新增订单页”进行现代化改造。引入卡片式布局 (Card Layout)、行内编辑 (Inline Editing) 以及更明确的视觉反馈 (User Feedback)，大幅提升系统可用性与专业质感。  
- ## 2. 需求详情：模块 1 - 订单商品明细页 (Order Item Details)
- ### 2.1 UI 布局重构 (Visual Hierarchy)
- **头部导航栏 (Header)**：
	- 左上角放置圆形的“⬅️ 返回台账”图标按钮（`flat round`），移除原有的文字拥挤感。
	- 居中或紧随其后显示清晰的大标题：`订单明细：[订单编号]`。
- **上下文信息卡片 (Context Card)**：
	- 在表格上方新增一个只读的卡片区域，展示该订单的核心上下文（发货人、收货人、总件数、总运费）。
	- 使用浅色背景（如 `bg-blue-50`）和网格布局（`ui.grid`），帮助用户在修改明细时了解全局状态。
- ### 2.2 核心交互：行内编辑 (Inline Editing)
- **废弃分离表单**：彻底移除页面底部的独立“修改”表单区域。
- **动态数据表格**：
	- 使用 `ui.table` 展示商品明细。
	- 通过注入 Quasar 的 `<template v-slot:body="props">` 插槽，将“品名、件数、重量、体积、计费方式、运费”等字段渲染为无边框的输入框 `<q-input dense borderless ...>`。
- **批量保存闭环**：
	- 在表格右上角放置主色调的“💾 保存所有修改”按钮。
	- 用户在表格内完成任意多行的修改后，点击一次保存，系统提取所有变更行发送至后端更新。
- ### 2.3 状态与反馈 (State & Feedback)
- 保存成功后，触发顶部居中的轻量级通知 `ui.notify('修改成功', type='positive')`。
- 数据保存时需有 `loading` 状态，防止重复点击。
- ## 3. 需求详情：模块 2 - 新增订单页 (Add Order Page)
- ### 3.1 模块化卡片布局 (Modular Card Layout)
  
  打破原有的自上而下的纯净流式布局，改用 Tailwind + Quasar Card 将长表单物理分隔为三大模块，降低认知负担：  
- **模块 A：基础与物流信息 (卡片 1)**
	- 包含：接单日期（`date_input`）、托运人（发货方）、收货人、目的地。
	- 采用两列网格布局 `ui.grid(columns=2)`，紧凑排列。
- **模块 B：财务与交接要求 (卡片 2)**
	- 包含：交接方式、付款方式、代收货款、回单要求。
	- 所有输入框统一采用 `props('outlined dense')`，提升现代感。
- **模块 C：商品明细清单 (卡片 3)**
	- **核心创新**：不再是单一文本框或多次提交，而是内嵌一个支持行内编辑的 `ui.table` 或基于 `ui.row` 动态生成的行列表。
	- 提供明显的 `➕ 新增商品行` 按钮。
	- 允许用户在一个页面内添加多条商品明细。
- ### 3.2 底部操作悬浮 (Floating Action)
- 页面最底部提供全宽或右对齐的主操作区。
- 按钮设计：“🚀 确认创建订单”，使用 `color='primary'`，并提供 `ui.notify` 成功反馈，完成后自动跳转回台账列表页。
- ## 4. 后端及数据层调整需求 (Backend Data Adaptations)
  
  为了支持前端的批量操作，需要在 `backend_db.py` 中进行相应的接口调整：  
- **新增批量更新函数 `update_order_item_batch(items_list: list[dict])`**：
	- 接收一个包含多个商品明细字典的列表。
	- 开启 SQLite 事务 (`BEGIN TRANSACTION`)，遍历更新对应的商品行数据。
	- 确保原子性：要么全部更新成功，要么全部失败回滚。
- **优化订单创建函数 `create_order_with_items(...)` (若不存在则新增)**：
	- 支持一次性接收“主订单数据”与“商品明细数组”。
	- 在一个事务中，先插入主订单获取 `order_id`，随后遍历插入商品明细，确保新增订单与明细的数据强一致性。
- ## 5. 验收标准 (Acceptance Criteria)
- **UI 一致性**：所有输入框统一为 `outlined dense` 风格，卡片有适度的阴影 (`shadow-sm` 或 `border`) 和内边距 (`p-4`)。
- **交互可用性**：
	- 在商品明细页可以直接点击单元格修改内容，点击“保存”后数据库正确更新且页面不需整页刷新即可显示最新值。
	- “返回台账”必须能正确保留之前的筛选状态（如果是 SPA 路由跳转需处理好状态传递）。
- **防呆与容错**：件数、重量等数值字段输入非法字符时，前端需拦截（使用 `type="number"`）；保存异常时弹出 `negative` 的错误提示。
