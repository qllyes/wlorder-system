# 中文字段转 SQL 字段规范工作流

本工作流旨在将业务口径提供的中文列名或字段名，规范化地翻译并转换为符合各类 SQL 数据库（MySQL, SparkSQL等）命名规范的字段名，并辅助生成数据字典或 DDL。

## 1. 命名规范原则
- **命名风格**：必须统一使用蛇形命名法（`snake_case`），即全小写字母加下划线分隔。
- **字符限制**：仅包含小写英文字母、数字和下划线（`a-z`, `0-9`, `_`），绝对不能以数字开头。
- **长度限制**：保证字段名长度在主流数据库的限制内（建议不多于 63 个字符）。
- **保留字/关键字避让**：严禁直接使用 SQL 关键字（如 `select`, `table`, `order`, `group`, `date`, `desc`, `key` 等）。如遇冲突，增加业务实体前缀或后缀（例如将 `order` 改为 `order_no` 或 `sales_order`，将 `date` 改为 `create_date`）。

## 2. 翻译与转换规则
- **优先级 1（行业通用缩写与惯用语）**：对于常见的业务和系统术语，优先使用业内约定俗成的英文缩写或全拼。
  - 数量 -> `qty` / `amount` / `cnt`
  - 编号/标识 -> `id` / `no` / `code`
  - 日期/时间 -> `date` / `time`
  - 状态 -> `status` / `state`
  - 备注 -> `remark` / `memo`
- **优先级 2（准确的英文翻译）**：使用贴切的英文单词进行直译，多个单词复合时自然使用下划线连接。
  - 出库时间 -> `outbound_time`
  - 供应商名称 -> `supplier_name`
- **优先级 3（拼音兜底，慎用）**：当遇到极其特殊的本土业务名词、公司内部特定代号，且极难找到准确英文表达时，允许使用拼音（建议全拼或首字母混合，需保证团队可读性），但应尽量避免。
  - 门店 -> `store`（绝对优先于 `mendian`）

## 3. 标准化输出格式 (字段映射表)
将翻译和校验后的字段以 Markdown 表格的形式输出，以便于用户核对。表格应包含如下列：
| 序号 | 中文字段名 | 推荐 SQL 字段名 | 数据类型建议（可选） | 翻译说明/备注 |
|---|---|---|---|---|
| 1 | 订单号 | `order_no` | VARCHAR(50) | 规避关键字，使用 no 作为后缀 |
| 2 | 客户名称 | `customer_name` | VARCHAR(100) | |
| 3 | 销售金额 | `sales_amount` | DECIMAL(15,2) | 涉及金额，建议 DECIMAL |

## 4. 辅助代码生成 (按需执行)
在用户确认字段映射表无误后，你可以询问并基于映射表快速生成：
1. **建表 DDL**（`CREATE TABLE ...`）。
2. **查询/插入视图所需的 AS 别名语句**（如 `SELECT 订单号 AS order_no, 客户名称 AS customer_name ...`）。

## 【执行步骤清单】
1. **接收输入**：获取用户提供的中文字段列表（文本、Markdown 表格或截图识别均可）。
2. **执行转换**：严格按照 [1. 命名规范] 和 [2. 转换规则] 处理列表中所有字段。
3. **输出对照**：打印字段映射表（参考第 3 节的格式）。
4. **生成代码**：根据用户的进一步指令生成建表 DDL 或 SQL 查询选择片段。
