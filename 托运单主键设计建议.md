# 托运单主键设计建议

> 目标：
> 1) 托运单使用“托运单号”作为主键；
> 2) 商品明细使用“托运单号 + 行号”作为联合主键；
> 3) 托运单号唯一、可追加、且能直观看出日期。

## 1. 推荐主键方案

### 1.1 托运单主键（waybill_no）

建议格式：`WBYYYYMMDDNNNN`

- `WB`：固定前缀（便于区分业务单号）
- `YYYYMMDD`：业务日期（例如 20260311）
- `NNNN`：当日顺序号，4 位左侧补零（例如 0001、0002）

示例：
- `WB202603110001`
- `WB202603110002`

这个格式满足：
- **唯一**：同一天内靠顺序号保证唯一；
- **追加**：顺序号单调递增；
- **可读日期**：一眼可见开单日期。

---

### 1.2 商品明细主键

建议在 `waybill_items` 表使用联合主键：

- `PRIMARY KEY (waybill_no, line_no)`

其中：
- `waybill_no`：所属托运单号
- `line_no`：行号，建议 `10, 20, 30...`（留出插行空间）

示例：
- `(WB202603110001, 10)`
- `(WB202603110001, 20)`

这样可以避免单独维护无业务意义的明细自增 ID，也更符合打印单“第几行”的业务表达。

## 2. 表结构建议（SQLite）

```sql
-- 主表：托运单
CREATE TABLE IF NOT EXISTS waybills (
  waybill_no      TEXT PRIMARY KEY,            -- WBYYYYMMDDNNNN
  waybill_date    TEXT NOT NULL,               -- YYYY-MM-DD（业务日期）
  customer_name   TEXT,
  customer_phone  TEXT,
  delivery_addr   TEXT,
  total_qty       INTEGER DEFAULT 0,
  total_weight_kg REAL DEFAULT 0,
  freight_fee     REAL DEFAULT 0,
  created_at      DATETIME DEFAULT (datetime('now','localtime'))
);

-- 子表：托运单商品明细
CREATE TABLE IF NOT EXISTS waybill_items (
  waybill_no      TEXT NOT NULL,
  line_no         INTEGER NOT NULL,            -- 10,20,30...
  product_name    TEXT,
  spec            TEXT,
  quantity        INTEGER DEFAULT 0,
  unit_weight_kg  REAL DEFAULT 0,
  line_weight_kg  REAL DEFAULT 0,
  PRIMARY KEY (waybill_no, line_no),
  FOREIGN KEY (waybill_no) REFERENCES waybills(waybill_no)
);
```

## 3. 单号生成机制（重点：并发安全）

不要在应用层“先查最大号再 +1”后直接写入，否则并发下会撞号。

建议增加“日序列表”：

```sql
CREATE TABLE IF NOT EXISTS waybill_daily_seq (
  biz_date TEXT PRIMARY KEY,                   -- YYYYMMDD
  seq      INTEGER NOT NULL
);
```

生成步骤（同一事务内）：

1. 取业务日期 `d = YYYYMMDD`
2. `INSERT INTO waybill_daily_seq(biz_date, seq) VALUES(d, 1)
   ON CONFLICT(biz_date) DO UPDATE SET seq = seq + 1`
3. 读取该日期最新 `seq`
4. 拼接成 `WB + d + LPAD(seq, 4, '0')`
5. 用这个 `waybill_no` 插入主表

> SQLite 可通过 `BEGIN IMMEDIATE` + 同事务更新序列表来确保并发下不重复。

## 4. 行号策略建议

- 初始新增明细按 `10,20,30...` 分配；
- 在两行中间插入时可用 `15`，避免全表重排；
- 如果长期频繁插入导致行号过密，可在“整理明细”时重排一次。

## 5. 与现有系统迁移建议（低风险）

如果当前使用的是 `shipment_id` 与 `shipment_products.id`：

1. 先给主表新增 `waybill_no`（唯一索引）；
2. 给子表新增 `line_no`；
3. 批量回填历史数据：
   - `waybill_no` 按历史 `created_at` + 当日序号生成；
   - `line_no` 按每单明细排序后赋值 `10,20,30...`；
4. 代码改为优先按新主键读写；
5. 稳定后再移除旧主键依赖。

## 6. 补充建议（实务）

- 增加 `UNIQUE (waybill_no)`（即使已主键，表达更清晰）；
- 打印模板里显示完整 `waybill_no`；
- 查询时支持按日期前缀搜索：`WHERE waybill_no LIKE 'WB20260311%'`；
- 若未来跨仓或多组织，可扩展为：`WB{仓码}{YYYYMMDD}{NNNN}`。

---

结论：
- 你这个场景最合适的就是 **“可读日期前缀 + 当日递增序号”** 的托运单号。
- 明细用 **`(托运单号, 行号)` 联合主键**，既稳定又符合业务操作与打印习惯。
